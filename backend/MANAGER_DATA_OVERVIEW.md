# Какие Данные Где Хранятся

Этот документ объясняет на бизнес-уровне, какие данные уже использует сервис, где они лежат и почему выбрана именно такая схема.

## Главная Идея

Одна пользовательская работа с одной 3D-моделью хранится как отдельный `run`.

`run` нужен как контейнер всей истории работы:
- кто загрузил модель;
- какой файл был загружен;
- на каком этапе находится обработка;
- какие результаты были получены;
- какие действия уже происходили.

За счёт этого система хранит не просто файлы, а полную сессию работы пользователя.

## Где Хранятся Данные

Сейчас данные разделены на два слоя:

1. `Postgres`
   Здесь лежат бизнес-данные, статусы, связи и метаданные.

2. `S3 / MinIO`
   Здесь лежат сами файлы.

Это сделано потому, что база данных хорошо подходит для структурированных данных и поиска, а файловое хранилище лучше подходит для тяжёлых объектов.

## Что Хранится В Postgres

В Postgres сейчас лежат:

### 1. Пользователи

Храним:
- email;
- хэш пароля;
- даты создания и обновления.

Зачем:
- чтобы пользователь мог войти в систему;
- чтобы все его данные были изолированы от других пользователей.

### 2. Runs

`run` это основная бизнес-сущность.

Храним:
- владельца `run`;
- название;
- текущий статус;
- ссылку на исходный файл;
- ссылку на выбранный коллаж в будущем;
- ссылку на последнюю генерацию в будущем;
- даты создания и обновления.

Зачем:
- чтобы видеть текущую стадию работы по модели;
- чтобы хранить одну полную сессию работы, а не разрозненные действия;
- чтобы пользователь мог возвращаться к своим задачам позже.

### 3. Artifacts

`artifact` это любой файл, связанный с `run`.

Сейчас уже используется для хранения исходного `.obj`.
Дальше в эту же сущность будут складываться:
- рендеры;
- коллажи;
- JSON-результаты.

Храним:
- тип файла;
- к какому `run` он относится;
- к какому пользователю относится;
- bucket;
- путь к объекту в S3;
- имя файла;
- content type;
- размер;
- checksum;
- технические метаданные;
- дату создания.

Зачем:
- чтобы не плодить отдельные таблицы под каждый тип файла;
- чтобы была единая модель работы со всеми результатами пайплайна;
- чтобы можно было легко расширять продукт.

### 4. Run Events

Это история действий внутри `run`.

Сейчас уже записываются события:
- `run_created`
- `source_uploaded`
- `render_started`

Зачем:
- чтобы понимать, что происходило с конкретной сессией;
- чтобы потом показывать историю пользователю;
- чтобы проще было разбирать ошибки и спорные ситуации.

### 5. Generations

Эта сущность уже заложена в модели данных, но пока ещё не используется в полном сценарии.

Она нужна для будущей логики, где система будет хранить каждую попытку генерации результата отдельно:
- какой вход использовался;
- какая модель запускалась;
- какой статус у генерации;
- была ли ошибка;
- какой файл получился на выходе.

Зачем:
- чтобы разделить сам файл-результат и факт бизнес-операции генерации;
- чтобы поддерживать несколько попыток генерации внутри одного `run`.

## Что Хранится В S3 / MinIO

В S3 хранится сам бинарный контент файлов.

Сейчас:
- исходные `.obj` файлы.

Позже:
- изображения рендеров;
- коллажи;
- итоговые JSON-файлы.

Зачем не хранить это в Postgres:
- файлы могут быть тяжёлыми;
- база данных не должна быть файловым архивом;
- S3-подход лучше масштабируется и проще для дальнейшей интеграции.

## Почему Это Хорошо Для Бизнеса

Такая схема полезна не только технически, но и продуктово:

- можно хранить полную историю работы пользователя;
- можно повторно запускать обработку на одном и том же `run`;
- можно добавлять новые этапы пайплайна без переделки всей модели;
- можно отделить пользовательские данные, бизнес-статусы и файлы;
- можно в будущем развивать сервис без перехода на микросервисы.

## Что Уже Работает

На текущий момент сервис уже умеет:
- создавать пользователя;
- авторизовывать пользователя;
- создавать `run`;
- загружать исходный `.obj` как `artifact`;
- переводить `run` в состояние обработки;
- сохранять историю ключевых событий.

## Что Появится Дальше

Следующие данные, которые логично появятся в системе:
- рендеры модели;
- коллажи;
- записи о генерации результата;
- итоговые JSON-артефакты;
- более подробные статусы пайплайна.

## Коротко

Если совсем коротко:

- `Postgres` хранит смысл и структуру процесса;
- `S3` хранит сами файлы;
- `run` это главная единица пользовательской работы;
- `artifacts` это все файлы внутри этой работы;
- `run_events` это история действий;
- `generations` это будущая история запусков inference.
