# Модель Данных Бэкенда Для MVP

## Основная Идея

Один `run` — это одна пользовательская сессия работы с одной CAD-моделью.

Внутри одного `run` хранятся:
- один исходный файл `obj`;
- несколько сгенерированных изображений;
- один или несколько коллажей;
- один или несколько сгенерированных JSON-файлов;
- история действий и статусов внутри этой сессии.

То есть `run` — это не просто файл и не один одноразовый прогон пайплайна, а контейнер для всех артефактов и событий, связанных с работой пользователя над одной моделью.

## Почему Такая Модель Подходит

Эта схема достаточно простая для MVP и при этом гибкая для будущих изменений:
- реализацию inference можно будет заменить позже;
- один и тот же `obj` можно прогонять несколько раз;
- сохраняется история генераций для каждого пользователя;
- микросервисы для этого не нужны.

## Рекомендуемые Сущности

### `users`

Хранит данные пользователя.

Предлагаемые поля:
- `id`
- `email`
- `password_hash`
- `created_at`

### `runs`

Один `run` = одна сессия пользователя над одной загруженной 3D-моделью.

Пользователь открывает сайт, загружает один `obj`, после чего создаётся `run`. Дальше внутри этого `run` он может много раз:
- генерировать рендеры;
- пересобирать коллажи;
- запускать генерацию JSON;
- сохранять и просматривать историю результатов.

Предлагаемые поля:
- `id`
- `user_id`
- `name`
- `status`
- `source_artifact_id`
- `selected_collage_artifact_id`
- `latest_generation_id`
- `created_at`
- `updated_at`

Примеры статусов:
- `created`
- `source_uploaded`
- `rendering`
- `rendered`
- `collages_ready`
- `generating_json`
- `completed`
- `failed`

### `artifacts`

Универсальная таблица для всех файлов. Для MVP это лучше, чем делать отдельные таблицы под рендеры, коллажи и JSON-файлы.

Предлагаемые поля:
- `id`
- `run_id`
- `user_id`
- `type`
- `bucket`
- `object_key`
- `file_name`
- `content_type`
- `size_bytes`
- `checksum`
- `meta_json`
- `created_at`

Возможные значения `type`:
- `source_obj`
- `render`
- `collage`
- `generated_json`

Что можно хранить в `meta_json`:
- угол рендера;
- индекс рендера;
- размер коллажа: `3`, `4` или `6`;
- версию промпта;
- ширину и высоту изображения;
- любые технические метаданные, которые могут меняться со временем.

### `generations`

Хранит каждую попытку генерации JSON как отдельную логическую запись.

Предлагаемые поля:
- `id`
- `run_id`
- `user_id`
- `input_collage_artifact_id`
- `output_artifact_id`
- `provider`
- `model_name`
- `prompt_version`
- `status`
- `error_message`
- `created_at`
- `completed_at`

Почему эта таблица нужна отдельно от `artifacts`:
- JSON-файл — это файловый артефакт;
- генерация — это бизнес-событие с метаданными модели, статусом и ошибками.

### `run_events`

Хранит историю шагов пайплайна и помогает в отладке.

Предлагаемые поля:
- `id`
- `run_id`
- `event_type`
- `payload_json`
- `created_at`

Примеры событий:
- `run_created`
- `source_uploaded`
- `render_started`
- `render_finished`
- `collage_created`
- `generation_started`
- `generation_failed`
- `generation_completed`

## Связи Между Сущностями

- один `user` имеет много `runs`;
- один `run` имеет много `artifacts`;
- один `run` имеет много `generations`;
- один `run` имеет много `run_events`;
- одна `generation` использует один коллаж как входной артефакт;
- одна `generation` создаёт один JSON как выходной артефакт.

## Минимальные Правила Для MVP

Чтобы первая версия оставалась простой, можно зафиксировать такие бизнес-правила:

- у одного `run` ровно один исходный `obj`;
- у одного `run` много рендеров;
- у одного `run` может быть несколько коллажей;
- у одного `run` может быть несколько генераций JSON;
- в таблице `runs` нужно хранить ссылки на текущий выбранный коллаж и на последнюю успешную генерацию JSON.

Такой подход даёт простой интерфейс для пользователя и при этом сохраняет историю всех действий внутри одной сессии.

## Что Хранить В Postgres И Что Хранить В S3

### Postgres

В Postgres хранить:
- пользователей;
- статусы `run`;
- метаданные всех файлов;
- метаданные генераций;
- историю шагов пайплайна;
- ссылки на объекты в S3.

### S3

В S3 хранить:
- исходные `.obj`;
- изображения рендеров;
- коллажи;
- сгенерированные JSON-файлы.

Важно: большие файлы не нужно хранить в Postgres. В базе должны лежать только метаданные и путь к объекту в S3.

## Практический Пример

Пользователь загружает одну модель.

Бэкенд создаёт:
- одну запись в `runs`;
- один артефакт типа `source_obj`;
- несколько артефактов типа `render` после рендера;
- один или несколько артефактов типа `collage`;
- одну запись в `generations`;
- один артефакт типа `generated_json` после inference;
- несколько записей в `run_events`.

После этого пользователь продолжает работать внутри того же `run`:
- может повторно сгенерировать рендеры;
- может собрать другой коллаж;
- может запускать генерацию JSON несколько раз;
- может видеть всю историю результатов в рамках одной сессии.

## Итоговая Рекомендация

Для стека `FastAPI + Postgres + S3` такая форма MVP выглядит удачно:

- один бэкенд-сервис;
- inference подключается через отдельный адаптерный интерфейс;
- `runs` используется как главная верхнеуровневая сущность;
- все файлы лежат в одной таблице `artifacts`;
- история коллажей и JSON-результатов сохраняется полностью для каждого пользователя.
